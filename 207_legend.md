### Часть 2. Эскалация

В 9:15 тимлид собрал экстренный созвон. Картина была такая:

- Репозиторий публичный — его может увидеть кто угодно
- Проект разрабатывался быстро, «чтобы работало» — security не приоритизировали
- CI/CD пайплайн проверяет только линтер и тесты — никаких security-сканеров
- Приложение уже задеплоено на тестовый сервер, доступный из интернета

CTO сказал фразу, которая определила план на ближайшие 90 минут:

> *«Нам нужно понять масштаб проблемы, устранить критическое и сделать так, чтобы это никогда не повторилось. Собирайте команду реагирования.»*

---

### Часть 3. Команда реагирования

**Вы** — члены команды реагирования на инциденты безопасности. Вас вызвали на экстренный security sprint.

**Ваша ситуация:**
- Репозиторий `ultra-vulnerable-repo` — простой REST API для управления задачами
- Написан на Python + FastAPI
- Работает с SQLite (в разработке) и PostgreSQL (в production)
- Упакован в Docker
- CI/CD через GitHub Actions — собирает, тестирует, деплоит
- **Ни одной security-проверки в пайплайне**

**Что известно:**
- В коде точно есть секреты
- В зависимостях — устаревшие пакеты с известными CVE
- SQL-запросы могут содержать инъекции
- Docker-образ не внушает доверия
- Есть админский endpoint с подозрительным функционалом

**Ваша миссия — три этапа:**

1. **Расследование (30 мин):** Найти ВСЕ уязвимости. Задокументировать. Оценить критичность.
2. **Устранение (15 мин):** Исправить критическое. Пересканировать. Подтвердить.
3. **Защита (30 мин):** Настроить автоматическую проверку в CI/CD, чтобы такое больше не прошло.

### Часть 4. Ставки

Через 48 часов — дедлайн от шантажиста. Но это не главное.

Главное — в репозитории могли быть и другие проблемы, которые ещё не нашли. Ваша задача — найти их **раньше**, чем это сделает кто-то другой.

Каждая непроверенная зависимость — потенциальная дыра. Каждый непроанализированный файл — возможный секрет. Каждый непрошедший security-этап в пайплайне — гарантия, что проблема уйдёт в production.
